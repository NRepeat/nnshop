<objective>
Make order creation resilient to partial failures and null cart delivery state, and block checkout during the post-login cart merge window.

Purpose: Three distinct gaps live in this plan. First (BUG-05): `cart.delivery.addresses.find(...)` at line 208 throws when `cart.delivery` is null — which happens for a cart that was just created or is mid-merge. Second (RELY-02): if the Shopify Admin API order creation succeeds but the subsequent `prisma.order.create` throws, the outer catch returns `{ success: false }` — the user sees an error even though their order exists in Shopify. Third (locked CONTEXT.md decision): the checkout button must be disabled with a spinner immediately after login while the cart merge is completing, so users cannot trigger a checkout while merge is in flight.

Note on RELY-02 retry clause: The RELY-02 requirement text includes "retry logic for idempotent operations." Application-level retry for `orderCreate` is descoped from this plan because: (1) the Shopify Admin API `orderCreate` mutation is NOT idempotent — retrying it would create duplicate orders; (2) transient HTTP errors (502/503) on the Admin API call are already handled by `AdminClient` at the HTTP layer; (3) wrapping a non-idempotent mutation in a retry loop is an anti-pattern that creates worse data integrity problems than the error it prevents. The isolated DB-save failure handling (Task 2) addresses the only retryable portion: the Prisma write after Shopify has already confirmed the order.

Output: A patched `create.ts` where `cart.delivery` is accessed via optional chaining and the Prisma DB save is wrapped in an isolated try/catch that logs on failure and returns success regardless. A patched `PaymentForm.tsx` that disables the checkout button for ~1.5s after login (optimistic merge window).
</objective>

1. Log with the required structured shape (for Phase 5 Sentry upgrade compatibility)
2. NOT re-throw — Shopify order already exists, user must see success

```typescript
// DB save is best-effort — Shopify order already exists and is confirmed
try {
  await prisma.order.deleteMany({
    where: { userId: session.user.id, draft: true },
  });
  await prisma.order.create({
    data: {
      shopifyOrderId: createdOrder.id,
      orderName: createdOrder.name,
      userId: session.user.id,
      draft: false,
    },
  });
} catch (dbError) {
  console.error('[create-order] DB save failed after Shopify success', {
    step: 'prisma-order-create',
    userId: session.user.id,
    orderId: createdOrder.id,
    error: dbError instanceof Error ? dbError.message : String(dbError),
  });
  // Do NOT re-throw — Shopify order exists; user must see success state
}

return {
  success: true,
  order: createdOrder,
};
```

The outer `catch (error)` block remains unchanged — it handles failures that occur before Shopify order creation (session missing, cart missing, API call failure). Only the DB save needs the isolated catch.

<task type="auto">
  <name>Task 3: Disable checkout button during post-login cart merge window (BUG-05 / locked decision)</name>
  <files>src/features/checkout/payment/ui/PaymentForm.tsx</files>
  <action>
This implements the locked CONTEXT.md decision: "Race condition (checkout triggered before merge completes): block checkout with a disabled button + spinner until merge finishes."

Cart merge runs server-side in the `onLinkAccount` hook — there is no direct client signal when it completes. The correct approach (per RESEARCH.md Open Question #2) is an optimistic timed disable: immediately after a session change (sign-in), disable the checkout submit button for 1500ms to cover the merge window.

In `PaymentForm.tsx`:

1. Import `useEffect` and `useState` from React if not already imported (check existing imports first).

2. Import `useSession` from the better-auth client. The auth client is at `src/features/auth/lib/auth-client.ts` — import the named `useSession` hook:
```typescript
import { useSession } from '@/features/auth/lib/auth-client';
```

3. Add state inside the component:
```typescript
const { data: session } = useSession();
const [isMerging, setIsMerging] = useState(false);
```

4. Add a useEffect that triggers the timed disable whenever the session user ID changes (i.e., after login):
```typescript
useEffect(() => {
  if (!session?.user?.id) return;
  setIsMerging(true);
  const timer = setTimeout(() => setIsMerging(false), 1500);
  return () => clearTimeout(timer);
}, [session?.user?.id]);
```

5. Find the checkout submit button in `PaymentForm.tsx`. It is likely rendered as a `<Button type="submit" ...>` or similar. Add `disabled={isMerging}` to the button props. If the button already has a `disabled` prop, use `disabled={isMerging || existingDisabledCondition}`.

6. Add a spinner indicator when `isMerging` is true. Use an inline conditional with the existing button children. If Lucide icons are already imported, add `Loader2` to the import and use `<Loader2 className="mr-2 h-4 w-4 animate-spin" />` before the button label text when `isMerging` is true:
```tsx
{isMerging && <Loader2 className="mr-2 h-4 w-4 animate-spin" />}
{isMerging ? t('syncingCart') : existingButtonLabel}
```

If `t('syncingCart')` key doesn't exist in `messages/`, use the existing button label text unchanged and only show the spinner (do not add a new i18n key in this task — the spinner alone satisfies the UX requirement).

Important: This is a purely additive change. Do NOT remove or restructure existing form logic, validation, or submission handlers. Only add the `isMerging` state, the `useEffect`, and the `disabled={isMerging}` + optional spinner to the button.
