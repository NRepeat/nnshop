<objective>
Harden the anonymous cart merge function to be resilient to partial failures.

Purpose: Currently `anonymousCartBuyerIdentityUpdate` swallows all errors silently, makes unguarded Prisma mutations outside a transaction, and has no retry logic — a failed Shopify API call leaves the database in inconsistent state. This fix ensures the DB is always consistent, failures are observable, and users get feedback when their cart can't sync.

Output: A rewritten `anonymous-cart-buyer-identity-update.ts` with Prisma transaction for all DB mutations, application-level retry with exponential backoff (3 attempts), full rollback on failure, structured step logging at every Shopify call boundary, single Shopify cart-lines fetch (not duplicated), and a sonner toast on total failure.
</objective>

```typescript
async function withRetry<T>(
  fn: () => Promise<T | null>,
  maxAttempts: number,
  baseDelayMs: number,

```

  <name>Task 2: Wrap DB mutations in Prisma transaction and implement retry+rollback</name>
  <files>src/entities/cart/api/anonymous-cart-buyer-identity-update.ts</files>
  <action>
Rewrite the `anonymousCartBuyerIdentityUpdate` main function body. Critical constraints from CONTEXT.md and research:

1. **Shopify calls happen BEFORE any Prisma transaction** — the transaction has a 5s default timeout; network I/O inside will exhaust it.
2. **Single Shopify fetch for anonLines** (PERF-02): fetch `anonLines` once at the top with `await getShopifyCartLines(anonCartRecord.cartToken)` and pass the result down. Do not re-fetch.
3. **Retry logic** (RELY-01): Wrap the `addLinesToCart` call in `withRetry(() => addLinesToCart(userCartToken, linesToAdd), 3, 300)`. The 3-retry with exponential backoff (300ms base = 300ms, 600ms, 900ms) is for the cart merge Shopify mutation specifically.
4. **Full rollback on failure** (RELY-01): If `addLinesToCart` returns null after all retries, do NOT proceed to the Prisma transaction. Log the failure and call `toast('Couldn\'t sync your cart. Your items are still saved.')` then return.
5. **Prisma transaction** (RELY-01): Wrap all `prisma.cart.update`, `prisma.cart.delete` calls inside `prisma.$transaction(async (tx) => { ... })`. Use `tx` not `prisma` inside. The two branches (user has cart / user has no cart) both need to be inside the transaction.
6. **Structured step logging** at every silent failure point in the main function body:

```typescript
console.error('[cart-merge] <description>', {
  step: '<step-slug>',
  userId: newUser.user.id,
  anonUserId: anonymousUser.user.id,
  orderId: undefined,
  error: error instanceof Error ? error.message : String(error),
});
```

Log specifically at: anon cart fetch failure, addLinesToCart total failure (before toast), updateShopifyBuyerIdentity failure (log but continue — buyer identity update failure is non-fatal), Prisma transaction failure.

For the toast: use `toast('Couldn\'t sync your cart. Your items are still saved.')` — this is the exact copy locked in CONTEXT.md. It is a neutral (not error) toast so users are not alarmed. Use `toast()` not `toast.error()`.

The overall structure should be:
```
1. Fetch anonCartRecord from DB
2. If none, return (no-op)
3. Fetch userCartRecord from DB
4. Fetch anonLines from Shopify (single fetch)
5. If userCart exists:
   a. If anonLines.length > 0: retry addLinesToCart up to 3 times
      - On total failure: log + toast + return (rollback = no DB changes yet)
      - On success: set finalCartToken
   b. updateShopifyBuyerIdentity (log failure but continue)
   c. prisma.$transaction: update userCart.cartToken, delete anonCart
6. Else (no userCart):
   a. updateShopifyBuyerIdentity (log failure but continue)
   b. prisma.$transaction: update anonCart.userId and cartToken
```

Note on Shopify partial merge (Pitfall 3 from research): if `cartLinesAdd` succeeded on attempt 1 but DB transaction later failed, Shopify already has the lines. Log with `step: 'shopify-merge-partial-success-db-rollback'` in the Prisma transaction catch block to document this accepted edge case.
