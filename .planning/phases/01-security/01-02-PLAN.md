<objective>
Install isomorphic-dompurify and apply HTML sanitization to the two components that render Shopify product HTML via dangerouslySetInnerHTML. Then verify Next.js Server Action CSRF protection is active, add documentation comments to key Server Action entry points, and demonstrate a rejected cross-origin POST via curl.

Purpose: Without DOMPurify, a Shopify product description containing `<script>` tags or event handler attributes would execute in the browser. The naive regex in ProductInfo.tsx only strips style attributes — it does not prevent XSS. CSRF protection is built into Next.js 14+ but must be verified active and documented so it is not accidentally disabled in the future.
Output: isomorphic-dompurify installed; two components sanitizing HTML; CSRF protection verified and documented in two Server Action files.

The `allowedDevOrigins` array that IS present is for cross-origin dev server access only — it does NOT affect Server Action CSRF protection. Document this distinction in a comment.

    Step 2 — Add CSRF documentation comment to src/entities/cart/api/anonymous-cart-buyer-identity-update.ts.
    At the very top of the file, before `'use server';`, add:
    ```typescript
    // SEC-02: CSRF PROTECTION VERIFIED — Next.js built-in Origin/Host header comparison
    // is active for this Server Action. On every invocation, Next.js compares the Origin
    // header to the Host header (or X-Forwarded-Host). Cross-origin requests are rejected
    // automatically (HTTP 403). No serverActions.allowedOrigins is configured in
    // next.config.ts, so only requests from the same origin (miomio.com.ua) are accepted.
    // Ref: https://nextjs.org/docs/app/guides/data-security#csrf-protection
    // Note: next.config.ts allowedDevOrigins controls dev server cross-origin access only;
    // it does NOT affect Server Action CSRF protection.
    ```

    Step 3 — Add the same CSRF documentation comment to src/features/order/api/create.ts.
    At the very top of the file, before its `'use server';` directive (or before the first import if 'use server' is not at the top — check the file first and place the comment at the top).

    Step 4 — Verify CSRF is active with a curl test. Run from the terminal:
    ```bash
    curl -s -o /dev/null -w "%{http_code}" \
      -X POST https://miomio.com.ua/api/auth/sign-in/email \
      -H "Origin: https://evil.example.com" \
      -H "Content-Type: text/plain;charset=UTF-8" \
      --data '{"type":"CALL_SERVER_ACTION"}'
    ```
    A response of 403, 405, or any non-2xx code confirms rejection. Document the actual response code in a comment added to the CSRF block in anonymous-cart-buyer-identity-update.ts:
    ```
    // Cross-origin test result: curl with Origin: https://evil.example.com → HTTP {actual_code}
    ```

    Note: The test URL uses the auth endpoint because it is publicly reachable without login. Any Server Action endpoint will exhibit the same CSRF behavior. If the production URL is not accessible from this environment, run against localhost:3000 with the dev server running instead, using `Origin: https://evil.example.com`.
